## 1. Конспект занятия (План для преподавателя)

### Часть 1. Гигиена репозитория (40 мин)

*Цель: Понять, что код и конфигурация живут раздельно.*

**Тезисы:**

1. **Code vs Config:** Код — это логика (она одинаковая везде). Конфиг — это настройки (база данных на лептопе и на проде — разные).
2. **12 Factor App:** Краткое упоминание методологии (пункт III: "Сохраняйте конфигурацию в среде выполнения").
3. **Опасность:** История о том, как разработчик выложил AWS-ключи на GitHub и получил счет на $10,000 за майнинг биткоинов через 15 минут.
4. **Инструментарий:**
    * `python-dotenv` — мост между файлом `.env` и `os.environ`.
    * `.gitignore` — фильтр для Git.

### Часть 2. Практика: "Чистый код" (60 мин)

*Цель: Настроить проект локально.*

**Практика:**

1. Создаем проект `secure-app`.
2. Пишем `app.py`, который падает с ошибкой, если не находит переменную `SECRET_KEY`.
3. Создаем `.env` (с секретом) и `.env.example` (пустой шаблон).
4. Настраиваем `.gitignore`:
    * Игнорируем `.env`, `__pycache__`, `venv/`.
5. **Pre-commit hooks:** Ставим `detect-private-key`, чтобы физически не дать себе сделать `git commit`, если внутри есть приватный ключ (из Урока 2) или явный секрет.

### Часть 3. GitHub и SSH (40 мин)

*Цель: Связать локальный Git с облаком.*

**Тезисы:**

* Зачем нужен удаленный репозиторий (Remote).
* Аутентификация: Почему пароли отменили и теперь только Token или SSH.

**Практика:**

1. Регистрируемся/Логинимся на GitHub.
2. **Связка с Уроком 2:** Берем публичный ключ (`cat ~/.ssh/id_ed25519.pub`), который генерировали на прошлом занятии, и добавляем его в настройки GitHub (Settings -> SSH and GPG keys).
3. Создаем **пустой** репозиторий на GitHub.
4. Пушим код: `git remote add origin ...` -> `git push`.

### Часть 4. Симуляция Деплоя (Deployment) (40 мин)

*Цель: Развернуть приложение на "сервере".*

**Сценарий:**

* Представим, что ваша локальная папка — это компьютер разработчика.
* Ваш Linux (WSL/VM) — это "Боевой Сервер" (Production).
* **Задача:** Запустить код на сервере.

**Алгоритм (демонстрируем на экране):**

1. Заходим на сервер (открываем терминал Linux).
2. `git clone git@github.com:user/repo.git`.
3. Пытаемся запустить -> Ошибка (нет зависимостей и нет `.env`).
4. Создаем виртуальное окружение: `python3 -m venv venv` -> `source venv/bin/activate`.
5. Ставим пакеты: `pip install -r requirements.txt`.
6. **Самое важное:** Создаем `.env` руками (`nano .env`) и вписываем туда секреты *для продакшна* (они могут отличаться от локальных!).
7. Запускаем -> Успех.

## 2. Домашнее задание №3: «Hello, Production!»

**Легенда:**
Вы написали прототип безопасного приложения. Теперь руководство требует развернуть его на тестовом сервере (вашем Linux окружении), чтобы проверить работу. Главное условие службы безопасности: **ни один пароль не должен засветиться в репозитории на GitHub**, но приложение на сервере должно работать и иметь доступ к секретам.

**Задание:**

### Этап 1: Разработка (Host/Локально)

1. Создайте папку проекта `safe_deploy`.
2. Напишите скрипт `main.py`, который:
    * Считывает переменную окружения `APP_SECRET` (используя `python-dotenv`).
    * Если переменная не найдена — выводит ошибку и завершается.
    * Если найдена — выводит: *"System started. Secret hash: [первые 3 символа секрета]***"*.
3. Создайте `.gitignore` (исключите `.env`, `venv`, `__pycache__`).
4. Создайте `.env.example` (пример конфига).
5. Инициализируйте Git, создайте репозиторий на GitHub и отправьте (push) туда код.
    * *Внимание:* Если в репозитории на GitHub окажется файл `.env` с реальным паролем — это **автоматический неуд** (0 баллов).

### Этап 2: Деплой (Linux/Server)

1. Откройте терминал вашего Linux (из Урока 1-2).
2. Клонируйте ваш репозиторий с GitHub (`git clone ...`).
3. Настройте окружение:
    * Создайте виртуальное окружение `venv`.
    * Установите зависимости (`pip install -r requirements.txt`).
4. **Конфигурация:**
    * Создайте файл `.env` внутри папки проекта на Linux (используя `nano` или `vim`).
    * Запишите туда значение `APP_SECRET=MySuperSecretProductionKey`.
5. Запустите приложение: `python3 main.py`.

**Формат сдачи:**

1. **Ссылка** на ваш репозиторий на GitHub (я проверю `.gitignore` и отсутствие секретов в истории коммитов).
2. **Скриншот** терминала Linux, на котором видно:
    * Команду запуска `python3 main.py`.
    * Вывод программы: *"System started. Secret hash: MyS***"*.
    * Вывод команды `ls -la`, подтверждающий наличие файла `.env` (которого нет на GitHub).

## 3. Критерии оценивания (Максимум 8 баллов)

| Критерий | Баллы | Описание |
| :--- | :---: | :--- |
| **Репозиторий (Git)** | **2** | Код залит на GitHub. В корне есть `.gitignore` и `.env.example`. История коммитов "чистая". |
| **Код (Python)** | **1** | Скрипт корректно обрабатывает отсутствие переменной окружения (не падает с Traceback, а пишет понятную ошибку) и использует `python-dotenv`. |
| **Безопасность (Secrets)** | **2** | Файл `.env` **отсутствует** в удаленном репозитории, но присутствует на скриншоте "сервера". |
| **Деплой (Linux)** | **2** | Предоставлен скриншот успешного запуска на Linux (видно приглашение командной строки bash/zsh, видно, что приложение подхватило секрет). |
| **Requirements** | **1** | В репозитории есть актуальный `requirements.txt`, позволяющий развернуть проект одной командой `pip install`. |
| **Итого** | **8** | |

### Штрафы (вычитаются из суммы)

* **-8 баллов (Обнуление):** Наличие файла `.env` с реальными ключами в публичном репозитории GitHub. Это грубейшее нарушение безопасности. Студент отправляется переделывать задание с созданием *нового* репозитория.
* **-2 балла:** Скриншот сделан в Windows (PowerShell/CMD), а не в Linux/WSL. Задача — симулировать сервер.
* **-1 балл:** "Грязный" репозиторий (закоммичена папка `venv` или `__pycache__`).
