**Тип занятия:** Воркшоп (Workshop)

**Сложность:** Средняя  
**Дедлайн по ДЗ:** 1 неделя (Задание небольшое, но важное для поддержки проекта).

## 1. Конспект занятия (План для преподавателя)

### Часть 1. Теория: Откуда берутся пакеты и проблемы? (40 мин)

*Цель: Разрушить миф о том, что Open Source — это всегда безопасно.*

**Тезисы:**

1. **Масштаб трагедии:** Современное приложение — это 10% вашего кода и 90% чужого (библиотеки). Если библиотека дырявая, ваше приложение тоже.
2. **Угрозы:**
    * **Typosquatting (Тайпосквоттинг):** Вы хотели `requests`, а написали `rquests` (злоумышленник залил пакет с похожим именем и вирусом).
    * **Dependency Confusion:** Когда публичный пакет подменяет внутренний корпоративный.
    * **Malicious Injection:** Взлом аккаунта разработчика популярной библиотеки и внедрение майнера (случай с `ua-parser-js`, `colors.js`).
    * **CVE (Known Vulnerabilities):** В старой версии Flask нашли дыру, а вы не обновились.

### Часть 2. Гигиена Pip и Фиксация версий (50 мин)

*Цель: Научиться "замораживать" состояние проекта.*

**Тезисы:**

1. **Семантическое версионирование (SemVer):** `MAJOR.MINOR.PATCH` (2.1.0).
2. **Проблема нечетких версий:**
    * В `requirements.txt` написано `FastAPI`. Pip поставит *последнюю*.
    * Завтра выйдет `FastAPI 1.0` (с ломающими изменениями) или `0.99` (с багом), и продакшн упадет.
    * **Правило:** В продакшене версии всегда фиксируются жестко (`==`).
3. **Виртуальное окружение (Deep Dive):**
    * Почему нельзя делать `pip install` глобально в систему (конфликты версий, права root).
    * Изоляция проектов.

**Практика:**

* Берем проект с прошлого урока.
* Смотрим `pip list`.
* Делаем "правильный" `pip freeze > requirements.txt`.
* Обсуждаем разницу между `package>=1.0` и `package==1.0`.

### Часть 3. Поиск уязвимостей: Pip-audit и Safety (60 мин)

*Цель: Использовать сканеры.*

**Инструменты:**

1. **CVE (Common Vulnerabilities and Exposures):** База данных известных уязвимостей.
2. **`pip-audit`:** Инструмент от Google/PyPA. Проверяет ваше окружение или файл requirements на наличие пакетов с известными CVE.
3. **`safety`:** Аналог (есть бесплатная версия, есть коммерческая).

**Практика (Live Hack & Fix):**

1. **Создаем проблему:** Специально устанавливаем дырявую версию популярной библиотеки.
    * Например: `pip install pyyaml==5.3` (в ней была критическая уязвимость десериализации).
    * Или `pip install django==2.2` (для примера).
2. **Сканируем:**
    * Запускаем `pip-audit`.
    * Видим красную таблицу: "Found 1 vulnerability". Читаем описание CVE.
3. **Исправляем:**
    * Обновляем пакет: `pip install pyyaml --upgrade`.
    * Фиксируем новую версию в `requirements.txt`.
    * Сканируем снова — чисто.

### Часть 4. Автоматизация в CI/CD (Pre-commit) (30 мин)

*Цель: Не пустить уязвимость в репозиторий.*

**Тезисы:**

* Человек забывает запускать аудит. Робот — нет.
* Добавляем `pip-audit` в наш любимый `pre-commit`.

**Практика:**

* Редактируем `.pre-commit-config.yaml`.
* Добавляем хук `pip-audit` (или локальный запуск команды).
* Теперь коммит не пройдет, если в `requirements.txt` есть дыры.

## 2. Домашнее задание №5: «Санитарный день»

**Легенда:**
Вы продолжаете работу над "Файловым менеджером". Перед тем как двигаться дальше, начальник безопасности (преподаватель) требует провести аудит зависимостей. Вам нужно доказать, что используемые библиотеки (FastAPI, Pydantic, Uvicorn) не содержат известных уязвимостей, и настроить автоматическую проверку.

**Задание:**

1. **Подготовка (Вредительство):**
    * В вашем проекте (`secure_start` или `file_manager`) временно установите заведомо уязвимый пакет. Рекомендую `insecure-package` (специальный тестовый пакет) или старую версию `pyyaml==3.13`.
    * Добавьте его в `requirements.txt`.

2. **Ручной аудит:**
    * Установите утилиту `pip-audit` (`pip install pip-audit`).
    * Запустите сканирование вашего файла зависимостей: `pip-audit -r requirements.txt`.
    * **Сделайте скриншот**, где видно найденную уязвимость (красный текст, номер CVE).

3. **Лечение:**
    * Удалите уязвимый пакет (или обновите его до безопасной версии, если он нужен проекту).
    * Обновите `requirements.txt` (командой `pip freeze > requirements.txt` или вручную, оставив только нужные: `fastapi`, `uvicorn`, `pydantic`, `python-dotenv`, `pip-audit`).
    * Убедитесь, что версии зафиксированы жестко (знак `==`).
    * Запустите `pip-audit` снова.
    * **Сделайте скриншот** успешного прохождения теста ("No known vulnerabilities found").

4. **Автоматизация (Pre-commit):**
    * Добавьте в `.pre-commit-config.yaml` хук для запуска `pip-audit`.
    * *Пример конфига:*

        ```yaml
        -   repo: https://github.com/pypa/pip-audit
            rev: v2.7.3  # (или актуальная версия)
            hooks:
            -   id: pip-audit
        ```

    * Если не получается подключить внешний репо (иногда бывает с pip-audit), можно использовать `local` хук:

        ```yaml
        -   repo: local
            hooks:
            -   id: pip-audit
                name: pip-audit
                entry: pip-audit
                language: system
                files: requirements.txt
        ```

    * Сделайте коммит изменений.

5. **Деплой на сервер (Linux):**
    * Закоммитьте изменения.
    * Зайдите на ваш учебный сервер (по SSH).
    * Сделайте `git pull`.
    * Обновите зависимости (`pip install -r requirements.txt`).
    * **Важно:** Обновите файл `.env` на сервере, если нужно (через `nano .env`).
    * Перезапустите приложение (Uvicorn).

**Формат сдачи:**
Ссылка на репозиторий + Markdown-файл (`audit_report.md`) с двумя скриншотами:

1. Скриншот с найденными уязвимостями.
2. Скриншот чистого аудита после исправления.

## 3. Критерии оценивания (Максимум 8 баллов)

| Критерий | Баллы | Описание |
| :--- | :---: | :--- |
| **Фиксация версий** | **1** | В `requirements.txt` все версии указаны жестко (через `==`), нет `*` или просто названий пакетов без версий. |
| **Демонстрация уязвимости** | **2** | Предоставлен скриншот работы `pip-audit`, где видна найденная проблема (CVE). Это доказывает, что студент умеет искать дыры. |
| **Чистый отчет** | **1** | Предоставлен скриншот, подтверждающий, что итоговый проект безопасен. |
| **Pre-commit hook** | **2** | В `.pre-commit-config.yaml` добавлен и настроен хук аудита. (Можно проверить в репозитории). |
| **Работа на сервере (Деплой)** | **2** | Приложение успешно развернуто на сервере (git pull), функционал работает удаленно. |
| **Итого** | **8** | |

### Штрафы

* **-2 балла:** В `requirements.txt` бардак (смешаны `==`, `>=`, и просто имена пакетов). Это потенциальная бомба замедленного действия.
* **-1 балл:** Лишние системные пакеты в `requirements.txt` (например, студент сделал `pip freeze` в глобальной среде, и туда попали `apt`-пакеты или системные утилиты Linux, не относящиеся к проекту).
* **-1 балл:** Отсутствие виртуального окружения (видно по путям в скриншотах).
