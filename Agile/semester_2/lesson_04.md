# Занятие №4: Фабричный метод (Factory Method)

**Цель занятия:** Изучить порождающий паттерн Factory Method. Студенты должны научиться выделять семейства схожих объектов в своих проектах и делегировать их создание специальным классам-фабрикам, соблюдая принцип OCP (Open-Closed Principle).

**Продолжительность:** 180 минут.

---

## Часть 1. Теория и Live Coding (90 минут)

### 1. Проблема создания объектов (15 мин)

* **Контекст:** В любой программе есть моменты, когда мы не знаем заранее, объект какого *конкретно* класса нам понадобится (какая кнопка в UI, какой враг на уровне, какой документ для печати).
* **Антипаттерн:** Использование оператора `new` внутри логики выбора (`switch/case`).
  * *Почему это плохо:* При добавлении нового типа придется переписывать существующий, отлаженный код.
* **Принцип OCP (Open-Closed):** Программные сущности должны быть открыты для расширения, но закрыты для модификации.

### 2. Паттерн Factory Method (20 мин)

* **Структура паттерна (UML):**
  * **Product (Абстрактный продукт):** Интерфейс или абстрактный класс (то, *что* создаем).
  * **Concrete Product:** Реализация (то, *как* это работает).
  * **Creator (Создатель):** Абстрактный класс с методом `CreateProduct()`.
  * **Concrete Creator:** Класс, который решает, какой именно продукт вернуть.

### 3. Live Coding: Система наград (Loot System) (55 мин)

*Демонстрация на нейтральном примере, который понятен всем, но вряд ли будет основой чьего-то проекта.*

* **Задача:** Реализовать сундук, из которого может выпасть разная награда (Золото, Зелье, Опыт), не зная заранее, что именно там лежит.
* **Шаг 1. Абстракция (Product):**

    ```csharp
    // Общий интерфейс для любой награды
    public abstract class Reward {
        public abstract void PickUp(); // Что происходит при поднятии
    }
    ```

* **Шаг 2. Конкретика (Concrete Products):**
  * `GoldReward`: пишет "Получено 100 золота".
  * `PotionReward`: пишет "Здоровье восстановлено".
* **Шаг 3. Фабрика (Creator):**

    ```csharp
    public abstract class RewardGenerator {
        // Фабричный метод
        public abstract Reward CreateReward();
        
        // Логика работы с продуктом (не знает тип награды!)
        public void OpenChest() {
            Reward reward = CreateReward();
            reward.PickUp(); 
        }
    }
    ```

* **Шаг 4. Конкретные фабрики:**
  * `GoldGenerator`: возвращает `new GoldReward()`.
  * `PotionGenerator`: возвращает `new PotionReward()`.
* **Финал:** Показываем, как клиентский код просто вызывает `generator.OpenChest()`, не заботясь о типе награды.

## Часть 2. Практическое задание (Домашняя работа) (90 минут + дом)

**Тема:** Внедрение Фабричного метода в свой проект.

### Задание

1. **Анализ проекта:**

    Посмотрите на архитектуру своей игры и найдите сущность, которая имеет несколько вариаций (минимум 2).
    * *Примеры:*
        * **RPG:** Враги (Орк, Эльф) / Оружие (Меч, Лук) / Заклинания (Огонь, Лед).
        * **Гонки:** Типы автомобилей (Спорткар, Грузовик) / Препятствия (Масло, Шипы).
        * **Стратегия:** Юниты (Пехота, Конница) / Здания (Казарма, Ферма).
        * **Квест:** Предметы инвентаря (Ключ, Записка).
2. **Реализация иерархии (Product):**
    * Создайте базовый класс или интерфейс для выбранной сущности (например, `Vehicle`, `Spell`, `Enemy`).
    * Создайте минимум **два** класса-наследника с разным поведением одного и того же метода (например, `Move()`, `Cast()`, `Attack()`).
3. **Реализация Фабрики (Creator):**
    * Создайте абстрактный класс-фабрику (например, `VehicleFactory`) с фабричным методом.
    * Создайте конкретные фабрики для каждого типа сущности (например, `SportCarFactory`, `TruckFactory`).
4. **Интеграция:**
    * Продемонстрируйте работу паттерна в вашем `GameManager` (или другом управляющем классе).
    * Создайте коллекцию фабрик или выберите одну конкретную фабрику, создайте объект через неё и вызовите его метод.

**Важно:** Код должен демонстрировать полиморфизм. Переменная, в которую вы сохраняете созданный объект, должна иметь тип **Базового класса**, а не конкретного.

**Дедлайн:** 1 неделя.

## Критерии оценивания (Максимум 8 баллов)

| Критерий | Баллы | Описание |
| :--- | :---: | :--- |
| **Выбор сущности** | **1** | Студент выбрал адекватную для своего проекта сущность. (Если студент делает гонки, а реализовал фабрику "Эльфов", которая нигде не используется — 0 баллов). |
| **Иерархия Продуктов** | **2** | Есть базовый класс/интерфейс и минимум 2 наследника. Реализована разная логика в переопределенных методах. |
| **Паттерн Factory Method** | **3** | Реализован класс Создатель (Creator) и Конкретные Создатели. Соблюдена сигнатура метода. |
| **Интеграция** | **1** | Паттерн реально используется в коде проекта (объекты создаются через фабрику, а не через `new` в `Main`). |
| **Качество кода** | **1** | Правильный нейминг, отсутствие мусора, работа через Git (ветка, PR). |
| **Итого** | **8** | |

### Штрафы

* **Нарушение абстракции:** `-2 балла`. Если метод фабрики возвращает конкретный класс (например, `public Orc Create()`), а не базовый (`public Enemy Create()`). Это ломает суть паттерна.
* **Switch вместо классов:** `-2 балла`. Если вместо иерархии фабрик сделан один класс с методом `Create(type)`, внутри которого `switch/case`. Это *Simple Factory*, а не *Factory Method*.
* **Бессмысленный код:** `-1 балл`. Если реализованные методы пустые или делают только `Console.WriteLine("Created")` без какой-либо логики самой сущности.

### Методический комментарий

* **Сложность:** Это занятие — фильтр. Те, кто не поймут здесь суть полиморфизма ("Зачем мне возвращать `Enemy`, если я создал `Orc`?"), будут страдать дальше.
* **Помощь:** Если студент не может придумать, что именно фабриковать в его игре — предложите универсальный вариант: **"События в игре"** (GameEvent). Событие "Начало уровня", "Получение урона", "Конец игры". Это подходит вообще всем жанрам.
* **Live Coding:** Во время демонстрации обязательно акцентируйте внимание на типе возвращаемого значения в методе фабрики. Это самая частая ошибка новичков.

Если этот вариант устраивает, переходим к **Занятию 5: Builder / Prototype и создание сложных объектов**
