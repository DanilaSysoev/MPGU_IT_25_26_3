# Занятие №5: Порождающие паттерны (Builder / Prototype)

**Цель занятия:** Изучить паттерны Builder и Prototype. Научиться создавать сложные составные объекты или эффективно клонировать существующие экземпляры.

**Продолжительность:** 180 минут.

---

## Часть 1. Теория и Live Coding (90 минут)

### 1. Проблема "Телескопического конструктора" (15 мин)

* **Контекст:** Представьте класс `Level` или `Character`. У него может быть 10-15 параметров (размер, биом, сложность, имя, раса, класс, начальное оружие...).
* **Антипаттерн:**

    ```csharp
    new Level(100, 100, Biome.Forest, Difficulty.Hard, true, false, null...);
    ```
  
  * *Проблема:* Читать невозможно. Легко перепутать `true` и `false`. Сложно добавлять новые параметры.

### 2. Паттерн Builder (Строитель) (30 мин)

* **Суть:** Отделяем конструирование сложного объекта от его представления.
* **Метафора:** Сборка бургера. "Мне, пожалуйста, булку, котлету, сыр, но БЕЗ огурцов".
* **Live Coding (Генератор отчетов/писем):**

    ```csharp
    // Сложный объект
    public class Report {
        public string Title;
        public string Body;
        public string Footer;
        public DateTime Date;
        // ...
    }

    // Строитель
    public class ReportBuilder {
        private Report _report = new Report();

        public ReportBuilder SetHeader(string title) {
            _report.Title = title;
            return this; // Fluent Interface!
        }
        
        public ReportBuilder SetContent(string text) {
            _report.Body = text;
            return this;
        }
        
        // ... и т.д.
        
        public Report Build() => _report;
    }

    // Использование
    var report = new ReportBuilder()
        .SetHeader("Otchet")
        .SetContent("Vse propalo")
        .Build();
    ```

* **Акцент:** Fluent Interface (`return this`) делает код читаемым как предложение на английском.

### 3. Паттерн Prototype (Прототип) (25 мин)

* **Суть:** Создание новых объектов путем копирования существующего (прототипа).
* **Проблема:** `new Orc()` создает "голого" орка. А нам нужен орк 5-го уровня с дубиной и в шлеме, как тот, что уже стоит на карте.
* **Live Coding (Клонирование монстра):**
  * Интерфейс `ICloneable` (встроенный в .NET) или свой метод `Clone()`.
  * **Важный нюанс:** *Shallow Copy* (поверхностное копирование) vs *Deep Copy* (глубокое копирование).
  * *Демонстрация:* Если у клона поменять оружие, а оно поменяется и у оригинала — это Shallow Copy (плохо). Нужно создавать *новый* экземпляр вложенных объектов.

### 4. Выбор паттерна (20 мин)

* Обсуждение: "Когда какой брать?"
  * *Builder:* Когда нужно пошагово собрать сложный объект с разными конфигурациями.
  * *Prototype:* Когда создание нового объекта "с нуля" слишком затратно или сложно, проще скопировать готовый.

## Часть 2. Практическое задание (Домашняя работа) (90 минут + дом)

**Тема:** Реализация сложного создания объектов.

### Задание

1. **Выбор задачи:**
    Проанализируйте свой проект. Вам нужно реализовать *один* из двух сценариев (на выбор):

    * **Сценарий А (Builder):** У вас есть сущность с множеством параметров, которую сложно создавать через конструктор.
        * *Примеры:* Генерация уровня (`LevelBuilder`), Создание персонажа (`CharacterBuilder`), Формирование сложного сообщения в лог (`LogEntryBuilder`).
    * **Сценарий Б (Prototype):** У вас есть сущность, которую нужно часто размножать с сохранением состояния.
        * *Примеры:* Спавн толпы врагов с одинаковыми статами (`Enemy.Clone()`), Размножение клеток/бактерий, Копирование предметов в инвентаре.

2. **Реализация (Builder):**
    * Создать класс-строитель.
    * Реализовать методы для настройки полей целевого объекта (минимум 3 метода настройки).
    * Реализовать метод `Build()` или `GetResult()`.
    * Желательно (но не обязательно) использовать Fluent Interface (цепочку вызовов).

3. **Реализация (Prototype):**
    * Реализовать интерфейс `ICloneable` или свой метод `Clone()` в выбранном классе.
    * Продемонстрировать **глубокое копирование** (если у объекта есть ссылочные поля, например, список предметов или ссылка на оружие — они тоже должны клонироваться).
    * Продемонстрировать создание клона и изменение его состояния (доказать, что оригинал не изменился).

4. **Интеграция:**
    * Использовать созданный паттерн в `GameManager` или логике инициализации.

**Дедлайн:** 1 неделя.

### Штрафы

* **Shallow Copy (для Prototype):** `-2 балла`. Если при клонировании списка предметов, оба объекта ссылаются на *один и тот же* список в памяти. Это грубая ошибка.
* **Бессмысленный Builder:** `-2 балла`. Если Builder просто дублирует конструктор (принимает все те же аргументы в своем конструкторе) и не дает гибкости.
* **Пустой метод:** `-1 балл`. Метод `Build()` возвращает `null` или `Clone()` бросает `NotImplementedException`.

### Методический комментарий

* **Prototype и C#:** В C# есть встроенный метод `MemberwiseClone()`. Он делает *поверхностную* копию.
  * *Важно:* Обязательно объясните студентам (или дайте ссылку), что `MemberwiseClone` копирует ссылки, а не объекты. Если у героя есть `List<Item> Inventory`, то клон будет иметь ссылку на *тот же самый* список. Добавление предмета клону добавит его и оригиналу. Это классические грабли, на которые они должны наступить.
* **Builder и Director:** В классике есть еще класс `Director` (Распорядитель), который управляет строителем. Для новичков это может быть оверхедом. Разрешите им использовать Builder напрямую в клиентском коде, без Директора, если логика не слишком сложная.
