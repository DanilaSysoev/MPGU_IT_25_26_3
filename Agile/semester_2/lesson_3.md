# Занятие №3: Паттерн Singleton и Игровой менеджер

**Цель занятия:** Изучить порождающий паттерн Singleton, понять разницу между статическим классом и синглтоном, реализовать центральный класс управления игрой (`GameManager`).

**Продолжительность:** 180 минут.

---

## Часть 1. Ретроспектива и Лекция (80 минут)

### 1. Паттерн Singleton (Одиночка) (30 мин)

* **Проблема:** В игре есть вещи, которые должны существовать в единственном экземпляре.
  * *Примеры:* Настройки звука, Подключение к базе данных, Сам игровой цикл (Game Manager).
  * *Если их будет два:* Конфликт ресурсов, рассинхронизация настроек.
* **Решение:** Класс, который сам контролирует создание своего единственного экземпляра.
* **Реализация (Live Coding):**

    ```csharp
    public class GameManager
    {
        private static GameManager _instance;

        // 1. Приватный конструктор! Чтобы никто не сделал new GameManager().
        private GameManager() { }

        // 2. Глобальная точка доступа.
        public static GameManager Instance
        {
            get
            {
                if (_instance == null)
                {
                    _instance = new GameManager();
                }
                return _instance;
            }
        }

        public void Run() { ... }
    }
    ```

* **Нюансы:**
  * *Lazy Initialization:* Объект создается только тогда, когда он впервые нужен.
  * *Thread Safety:* Упомянуть `lock` или `Lazy<T>`, но сказать, что для нашей однопоточной консольной игры простая реализация допустима.

### 2. Singleton vs Static Class (15 мин)

* **Вопрос студентов:** "Зачем писать столько кода, если можно просто сделать `static class Game`?"
* **Ответ:**
    1. **Наследование:** Синглтон может наследовать интерфейсы (`IGameManager`), статический класс — нет.
    2. **Передача:** Синглтон можно передать в метод как параметр, статический класс — нет.
    3. **Управление жизнью:** Синглтон можно уничтожить (присвоить null) и пересоздать, статику выгрузить из памяти сложно.

### 3. Темная сторона (Anti-pattern) (15 мин)

* Почему сеньоры не любят Singleton?
  * *Глобальное состояние:* Любая часть кода может изменить состояние синглтона, и вы не найдете, кто это сделал.
  * *Тестирование:* Unit-тесты должны быть изолированы. Если один тест изменил Синглтон, второй тест может упасть.
* **Вывод:** Используем аккуратно. Для `GameManager` в учебном проекте — можно. Для `Player` — категорически нельзя (вдруг захотим мультиплеер?).

## Часть 2. Воркшоп / Практическое задание (100 минут)

**Тема:** Реализация Game Manager.

### Легенда

Архитектура утверждена. Теперь нужно создать "Сердце" игры — менеджер, который будет управлять запуском, настройками и выходом.

### Задание пошагово

1. **Создание ветки:**

    * `git checkout -b feature/game-manager`

2. **Реализация Singleton:**
    * Создать класс `GameManager`.
    * Реализовать паттерн Singleton (приватный конструктор, поле instance, свойство Instance).
3. **Перенос логики:**
    * Перенести цикл `while(true)` из `Program.cs` в метод `Run()` класса `GameManager`.
    * В `Program.cs` оставить только одну строчку (или две):

        ```csharp
        GameManager.Instance.Run();
        ```

4. **Добавление состояния (настроек):**
    * Добавить в `GameManager` настройки игры. Выберите характерные для вашего проекта настройки, например:
        * `MapWidth` (int)
        * `MapHeight` (int)
        * `Difficulty` (enum)
    * Эти данные должны задаваться при первом обращении или иметь дефолтные значения.
5. **Демонстрация работы:**
    * В методе `Run()` вывести приветствие, например: "Game Started with difficulty: [Difficulty]".

**Дедлайн:** 1 неделя.

### Штрафы (Вычитаются из суммы)

* **Public Constructor:** `-3 балла`. Это убивает суть паттерна. Грубая ошибка.
* **Статический класс:** `-2 балла`. Если вместо Singleton сделан `static class GameManager`. Задача была именно на паттерн.
* **Грязный Program.cs:** `-1 балл`. Если в `Program.cs` осталась логика инициализации переменных, которая должна быть в Менеджере.
